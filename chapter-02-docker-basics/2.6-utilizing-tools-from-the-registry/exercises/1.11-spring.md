# Exercise 1.11 - Spring

## Objective

Learn how to translate standard developer documentation (a README file) into a functional `Dockerfile` by leveraging pre-built language runtime images.

---

## Concepts Practiced

- Pre-built Language Images (e.g., `amazoncorretto`)
- Translating README instructions to Dockerfile syntax
- Dependency Management in Java (Maven/Gradle)
- Building and Running custom images

---

## Exercise Summary

The task involves taking an existing, slightly older Java Spring project and containerizing it. Instead of starting with a bare OS image and writing complex bash scripts to install the correct version of Java, the exercise encourages finding and using an official, pre-built Java image from Docker Hub. The required build steps and execution commands must be deciphered from the project's README and codified into the `Dockerfile`.

---

## Hints

- **Base Image:** Search Docker Hub for `amazoncorretto` (or a similar official Java image) and pick a tag that matches the Java version required by the project.
- **Build vs Run:** The README will likely instruct you to build the project first (e.g., using a wrapper script like `./mvnw`) before running the resulting `.jar` file. Both of these steps need to be translated into your Dockerfile.
- **Exposing Ports:** Check the README to see which port the Spring application listens on, and make sure to publish it when running the container.

## Verification

### Expected Result

After building the image and running the container with the correct port published, navigating to `http://localhost:8080` (or whichever port the app uses) in a web browser should display a 'Success' message generated by the Spring application.

---

## Challenges Faced

- Identifying the correct base image tag that satisfies the specific legacy Java version requirements of the older Spring project.

---

## Lessons Learned

- **The Power of Pre-built Images:** Using a pre-built language image eliminates hours of frustrating configuration errors related to installing language runtimes and setting up environment variables (like `JAVA_HOME`).
- **Documentation is Universal:** If a project has a well-written README detailing how to run it locally, containerizing it is usually a simple 1-to-1 translation process into `Dockerfile` instructions.
