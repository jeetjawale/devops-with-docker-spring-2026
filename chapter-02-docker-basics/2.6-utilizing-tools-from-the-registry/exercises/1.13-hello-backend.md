# Exercise 1.13 - Hello backend

## Objective

Containerize a Go (Golang) backend service and manage potential processor architecture mismatches during the build process.

---

## Concepts Practiced

* Go Application Containerization
* Port Publishing (`-p 8080:8080`)
* Cross-Platform Emulation (`--platform`)

---

## Exercise Summary

This is the second part of the full-stack project. The task is to containerize a provided backend application written in Go. The challenge involves reading the provided documentation to determine how to build and run the Go binary inside the container. Additionally, the exercise introduces the complication of architecture mismatches—specifically, ensuring the image can be built and run correctly on ARM-based machines (like Apple Silicon M-series Macs) if necessary.

---

## Hints

* **Base Image:** Unlike the frontend exercise, there are no strict restrictions against using a pre-built language image here. Starting with an official `golang` image makes the process much simpler.
* **Compilation:** Go is a compiled language. You will need to run a build command (like `go build`) inside the `Dockerfile` to create the executable binary, and then set your `CMD` to execute that resulting binary.
* **M1/M2 Mac Users:** If you encounter weird execution errors, use the `docker build --platform linux/amd64` flag to force Docker to build and emulate the x86 architecture.

---

## Commands Used

```bash
# Commands explored while solving
docker build -t backend .
docker run -p 8080:8080 backend
```

---

## Verification

### Expected Result

Running the container and publishing its internal port 8080 to the host should make the API accessible. Navigating to `http://localhost:8080/ping` in a web browser (or using `curl`) should return a simple text response: "pong".

---

## Challenges Faced

* Understanding the Go compilation step and ensuring the resulting binary had the correct permissions to execute as the `CMD`.

---

## Lessons Learned

* **Compiled vs Interpreted:** Containerizing a compiled language like Go requires a build step that generates a static binary. (Later on, this allows for incredibly small images using multi-stage builds).
* **Architecture Matters:** As the industry transitions from purely x86_64 to a mix of x86 and ARM (AWS Graviton, Apple Silicon), being aware of the `--platform` flag is critical for ensuring images run consistently everywhere.
