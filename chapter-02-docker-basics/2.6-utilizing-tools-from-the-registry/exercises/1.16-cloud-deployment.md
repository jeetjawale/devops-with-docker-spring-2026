# Exercise 1.16 - Cloud deployment

## Objective

Prove the ultimate value of containerization by deploying a locally built Docker image to a public cloud Platform-as-a-Service (PaaS) provider, making it accessible over the internet.

---

## Concepts Practiced

* Cloud Deployment (PaaS)
* Container Portability
* Public Accessibility

---

## Exercise Summary

The final exercise of the part. The task is to take any containerized web application (either one from a previous exercise or a personal project) and deploy it to a cloud provider such as Render, Heroku, or Fly.io. This demonstrates that an image built and tested locally will run identically in a production cloud environment.

---

## Hints

* **Provider Selection:** Choose a provider that natively supports deploying Docker containers. Many PaaS providers allow you to simply connect your GitHub repository containing the `Dockerfile`, or provide a direct link to the image you published on Docker Hub.
* **Port Binding Constraints:** Cloud providers often dynamically assign the port your application must listen on via an environment variable (usually `$PORT`). Ensure your application or Dockerfile is configured to respect this variable rather than hardcoding a specific port (like 8080), or use the provider's specific Docker deployment documentation to map it correctly.

## Challenges Faced

* Adapting the application to listen on the specific port injected by the cloud provider's environment variables, rather than the hardcoded local port.

---

## Lessons Learned

* **True Portability:** "It works on my machine" translates perfectly to "It works in the cloud" when using Docker. The deployment process is drastically simplified because the cloud provider doesn't need to know anything about the language or dependencies—it just runs the container.
* **Environment Configuration:** Cloud deployments heavily rely on environment variables (`ENV`) to inject specific configurations (like database URLs or port numbers) into the container at runtime.
