# Exercise 1.12 - Hello frontend

## Objective

Containerize a Node.js frontend application strictly using a bare OS base image to understand the manual steps required to configure a runtime environment.

---

## Concepts Practiced

- Manual Runtime Installation (Node.js via `apt-get`)
- Frontend Containerization
- Port Publishing (`-p 5001:5001`)
- `EXPOSE` instruction

---

## Exercise Summary

This is the first part of a larger project. The task is to containerize a provided frontend application. Unlike the previous exercise where a pre-built language image was used, this exercise strictly mandates using the `ubuntu` base image. This means the `Dockerfile` must include the manual shell commands required to download and install the correct version of Node.js before the application code can be copied, built, and executed.

---

## Hints

- **Installing Node:** You will need to find the specific terminal commands to install Node.js on Ubuntu (usually involving `curl` and adding a NodeSource repository) and put them in a `RUN` instruction.
- **Dependencies:** After installing Node, you must copy the package manifest and run the dependency installation command (e.g., `npm install`).
- **Port Mapping:** The application runs on port 5001. Ensure you expose this in the Dockerfile and publish it using the `-p` flag when running.

---

## Commands Used

```bash
# Commands explored while solving
docker build -t frontend .
docker run -p 5001:5001 frontend
```

---

## Verification

### Expected Result

After the container starts, it will take a few seconds to print "Accepting connections at http://localhost:5001". Once printed, navigating to that address in a local web browser should display the frontend interface successfully.

---

## Challenges Faced

- Figuring out the correct, non-interactive `apt-get` and `curl` commands to cleanly install Node.js inside the Ubuntu base layer without user prompts stalling the build.

---

## Lessons Learned

- **Appreciation for Pre-built Images:** Manually writing the installation steps for a language runtime is tedious and error-prone. It highlights exactly why images like `node:18` are so heavily utilized in the industry.
- **Patience with Webpack/React:** Frontend build tools often take a few seconds to compile and bind to a port when starting up, even inside a container.
