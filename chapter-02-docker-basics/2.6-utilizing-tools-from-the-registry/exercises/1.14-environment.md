# Exercise 1.14 - Environment

## Objective

Configure two independent containers (a frontend and a backend) to communicate with each other by utilizing environment variables, while understanding the networking context of browser-executed code.

---

## Concepts Practiced

* Environment Variables (`ENV`)
* Inter-Container Communication Architecture
* Cross-Origin Resource Sharing (CORS)
* Browser vs Container Networking Contexts

---

## Exercise Summary

The final part of the local project setup. The task is to run both the frontend (from 1.12) and backend (from 1.13) containers simultaneously. However, the frontend needs to know where to send its API requests. This configuration must be injected into the frontend `Dockerfile` using the `ENV` instruction. A critical realization required here is that the frontend JavaScript code runs in the *user's local browser*, not inside the frontend container, which changes how the environment variable URL must be formatted.

---

## Hints

* **The Networking Context Trap:** Because the frontend is just serving static JavaScript files that are downloaded and executed by your local web browser, the API requests originate from your browser. Therefore, the backend URL provided via `ENV` must point to your *host machine's* mapped port (e.g., `http://localhost:8080`), NOT the internal Docker network IP of the backend container.
* **Setting ENV Variables:** Check the frontend README to find the exact name of the environment variable expected (e.g., `REACT_APP_BACKEND_URL`), and define it in the frontend `Dockerfile` using `ENV <key>=<value>`.
* **CORS:** Ensure the backend is configured (via its own environment variables, if necessary, based on its README) to accept cross-origin requests from the frontend's origin (e.g., `http://localhost:5001`).

## Commands Used

```bash
# Commands explored while solving

docker run -p 5001:5001 frontend
docker run -p 8080:8080 backend
```

With both containers running in separate terminals, navigating to the frontend at `http://localhost:5001` and clicking the specific button for Exercise 1.14 should trigger a successful network request to the backend. The button should turn green, indicating the frontend received the "pong" response from the backend API.

## Challenges Faced

* Initially struggling with CORS errors in the browser's developer console because the backend was rejecting requests from the frontend's origin.
* Understanding why setting the frontend's environment variable to an internal Docker IP failed (because the browser on the host machine couldn't resolve it).

---

## Lessons Learned

* **Execution Context:** The most important lesson is differentiating between where code is hosted (the frontend container) and where it actually executes (the browser). This dictates how URLs and network paths must be structured.
* **Environment Variables:** Injecting configuration via `ENV` makes images reusable across different environments (local, staging, production) without altering the source code.
