# Exercise 1.9 - Volumes

## Objective

Learn how to use bind mounts to extract and persist data generated internally by a container out to the local host machine's filesystem.

---

## Concepts Practiced

* Bind Mounts (`-v` flag)
* Data Persistence
* File-level Mounting
* Preventing Ephemeral Data Loss

---

## Exercise Summary

The provided application image behaves as a background worker, generating timestamp logs and writing them to a specific internal file (`/usr/src/app/text.log`). By default, these logs would be trapped inside the container and permanently lost when the container stops. The task is to start the container while explicitly mapping that internal log file directly to a local file on the host machine using a volume flag, allowing real-time monitoring and persistence of the logs.

---

## Hints

* **File vs Directory Mounting:** The volume flag (`-v`) allows you to mount either entire directories or specific individual files. In this scenario, mounting the specific file is required.
* **The "Empty Folder" Trap:** If you instruct Docker to bind mount a specific file path on your host, but that file *does not actually exist* on your host yet, Docker's default behavior is to assume you meant to map a directory. It will create an empty folder with that name instead of a file, which will break the application trying to write text to it.
* **Preparation:** It is highly recommended to manually create an empty log file on your local machine (e.g., using `touch text.log`) *before* running the container command to avoid the directory trap.
* Use absolute paths or the `$(pwd)` shell shortcut to ensure the mount maps exactly where you expect.

## Verification

### Expected Result

After launching the container with the correct bind mount, the application should run normally. You should immediately see the local log file on your host machine begin populating with timestamps every two seconds. You can verify this by running `tail -f text.log` locally on your host.

---

## Challenges Faced

* Getting the source and destination paths correct, and ensuring the local file existed beforehand to prevent Docker from creating a directory instead of mounting a file.

---

## Lessons Learned

* **State Persistence:** Containers are designed to be stateless and disposable. Any data that you care about keeping (like databases, logs, or user uploads) *must* be mapped out to the host via a volume or bind mount.
* **Precise Mapping:** You have granular control over what you expose from the host to the container. You don't have to mount the entire current directory; you can surgically mount only the exact files the application needs.
