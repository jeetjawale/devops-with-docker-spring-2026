# Exercise 1.10 - Ports open

## Objective

Understand how to expose an internal containerized web service to the outside world by actively publishing its ports to the local host machine.

---

## Concepts Practiced

- Port Publishing (`-p` flag)
- Container Networking Isolation
- Web Service Accessibility
- Host to Container Mapping

---

## Exercise Summary

The exercise involves running an image that hosts a web server. The server is configured internally to listen for HTTP traffic on port `8080`. By default, this network interface is entirely isolated within the container, making the web app unreachable from the host machine's browser. The task is to use the port publishing flag to drill a hole through this isolation, explicitly mapping the local machine's port `8080` to the container's internal port `8080`, and verifying the connection via a web browser.

---

## Hints

- **The Command Flag:** Remember the specific flag used during `docker run` to define port mappings.
- **Syntax Order:** The mapping syntax always expects the outside host port first, followed by the inside container port (`host-port:container-port`). Reversing them will result in a failure to connect.
- **Arguments:** If you are using the base `devopsdockeruh/simple-web-service` image directly, don't forget that it requires the argument `server` to actually start the web app. Alternatively, if you use the custom image you built in Exercise 1.8, the `CMD` handles this argument automatically.

## Verification

### Expected Result

After running the container with the correct port mapping, you should be able to open a standard web browser on your host machine, navigate to `http://localhost:8080`, and immediately see the JSON response generated by the containerized application.

---

## Challenges Faced

- Remembering that exposing a port via an `EXPOSE` instruction in a `Dockerfile` is not enough to actually route traffic; the `-p` runtime flag is mandatory for accessibility.

---

## Lessons Learned

- **Network Isolation:** By default, containers are incredibly secure from a networking perspective because they share nothing with the host. You have to explicitly opt-in to open communication channels.
- **Port Flexibility:** You don't have to map to the exact same port number. If port `8080` was already taken on the host machine by another app, you could easily map `-p 3000:8080` to make the containerized app accessible at `localhost:3000` instead.
