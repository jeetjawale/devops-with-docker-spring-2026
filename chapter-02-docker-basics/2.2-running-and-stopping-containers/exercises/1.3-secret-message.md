# Exercise 1.3 - Secret message

## Objective

Learn how to connect to an already running container and execute secondary commands inside it to inspect its internal state or filesystem.

---

## Concepts Practiced

* Background Execution (Detached Mode)
* Secondary Process Execution
* Container Internal Filesystem Inspection
* Log Tailing (`tail -f`)

---

## Exercise Summary

The task involves running a specific provided image that continuously generates logs and writes them to a file stored inside the container's isolated filesystem. The objective is to keep the container running in the background, open a new interactive session into it, and follow the log file using the `tail` command to discover a secret message that appears periodically.

---

## Hints

* Start the container in detached mode so it runs in the background. This keeps your terminal free.
* You need to use the Docker command designed to run a *new* process inside an existing container, rather than starting a new container.
* When executing the new process, make sure to use the flags that provide an interactive terminal (e.g., `-it`) so you can explore the filesystem and run the `tail -f` command.

---

## Commands Used

```bash
# Commands explored while solving
docker run -d
docker exec -it
tail -f
```

---

## Verification

### Expected Result

After successfully jumping into the running container and executing the `tail` command on the correct log file, the terminal should pause and wait. Within 10 seconds, the secret message should be printed to the screen.

---

## Challenges Faced

* Remembering the difference between `docker attach` (which connects to the primary process running the app) and `docker exec` (which allows you to run a new command like `bash` or `sh` to poke around).

---

## Lessons Learned

* Containers behave very much like independent virtual machines when you `exec` into them; you can navigate their internal directory structure freely without disturbing the main application running on PID 1.
* Reading files directly from the container's filesystem is highly useful when applications log to internal files rather than standard output (STDOUT).
