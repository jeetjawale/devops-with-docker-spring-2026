# Exercise 1.4 - Missing dependencies

## Objective

Understand how to interact with a running container to troubleshoot and resolve missing dependency errors by installing packages manually via the command line.

---

## Concepts Practiced

* Interactive Shells (`-it`)
* Package Management inside Containers (`apt-get`)
* Debugging Running Processes
* Ephemeral Container State

---

## Exercise Summary

This exercise simulates a common real-world debugging scenario. An interactive script is run inside a fresh Ubuntu container. The script asks for a website URL and attempts to fetch it using `curl`. Because base Ubuntu images are stripped down to save space, `curl` is not installed by default, causing the script to fail. The task is to leave the script running in one terminal, open a separate terminal session into the same container, and manually install the missing tool so the script can succeed.

---

## Hints

* The primary script must remain running in the first terminal. Use a separate terminal window to fix the issue.
* Use `docker exec` to start a new, interactive `bash` session inside the problematic container.
* Since the base image is Ubuntu, you will need to use standard Debian/Ubuntu package manager commands to install the missing tool. Don't forget that you usually have to update the package lists before installing a new tool.

---

## Commands Used

```bash
# Commands explored while solving
docker run -it
docker exec -it
apt-get update
apt-get install
```

---

## Verification

### Expected Result

Initially, the script will output `curl: not found`. After successfully installing the dependency via the second terminal, providing a URL to the script in the first terminal should successfully fetch and print the raw HTML response of the requested website (e.g., `301 Moved Permanently`).

---

## Challenges Faced

* Remembering that package lists must be updated (`update`) before a package can be installed (`install`), otherwise the package manager won't be able to find the requested software.

---

## Lessons Learned

* **Minimal Base Images:** Official base images (like `ubuntu` or `alpine`) are purposefully minimal. Standard utilities you expect to be present (like `curl`, `ping`, or `nano`) are often omitted to keep image sizes small and reduce the security attack surface.
* **Ephemeral Changes:** While using `docker exec` to manually install packages is excellent for quick debugging, these changes are temporary. If the container is stopped and removed, the installed packages are permanently lost. To make changes permanent, they must be written into a `Dockerfile`.
