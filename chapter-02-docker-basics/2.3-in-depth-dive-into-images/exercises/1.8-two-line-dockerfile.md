# Exercise 1.8 - Two line Dockerfile

## Objective

Understand the interaction between `ENTRYPOINT` and `CMD` instructions within a `Dockerfile` and how to supply default arguments to an existing image.

---

## Concepts Practiced

* `Dockerfile` Inheritance
* `FROM` instruction
* `CMD` instruction (as default arguments)
* `ENTRYPOINT` interaction

---

## Exercise Summary

A provided third-party image uses the `ENTRYPOINT` instruction to define the main application it runs. However, this application requires a specific command-line argument (e.g., "server") to actually start the web service; otherwise, it fails or does nothing. The task is to create a very minimal, two-line `Dockerfile` that inherits from this third-party image and uses the `CMD` instruction to seamlessly provide the missing argument by default. 

---

## Hints

* **Inheritance:** Your `FROM` statement should point directly to the provided `devopsdockeruh/simple-web-service:alpine` image.
* **CMD as Arguments:** When an image defines an `ENTRYPOINT`, anything provided via `CMD` acts as default arguments appended to that entrypoint. 
* The exercise title literally says "Two line Dockerfile"—don't overcomplicate it. You only need `FROM` and `CMD`.

## Verification

### Expected Result

Building the new two-line Dockerfile and running the resulting image should immediately output console logs indicating that a web server (using GIN) has started and is listening on port 8080. You shouldn't need to append any arguments manually to your `docker run` command.

---

## Challenges Faced

* Understanding the conceptual difference between `CMD` acting as the command itself versus `CMD` acting as an argument list when `ENTRYPOINT` is present.

---

## Lessons Learned

* **Layering Images:** You don't have to start from an OS like Ubuntu or Alpine every time. You can use another fully-featured application image as your base and just tweak its configuration or default behavior.
* **ENTRYPOINT vs CMD:** 
  * `ENTRYPOINT` defines the strict executable that *must* run.
  * `CMD` provides the default arguments for that executable, which the user can easily override at the command line (e.g., `docker run my-image override-arg`).
