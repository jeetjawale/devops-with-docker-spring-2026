# Exercise 1.7 - Image for script


## Objective

Learn how to automate environment setup by writing a `Dockerfile` that packages a custom script along with all its required dependencies into a single, reproducible image.

---

## Concepts Practiced

* `Dockerfile` Creation
* `FROM`, `RUN`, `COPY`, `CMD` instructions
* Automated Dependency Installation
* Image Building (`docker build`)

---

## Exercise Summary

In a previous exercise, we had to manually intervene in a running Ubuntu container to install `curl` because a script required it. This exercise solves that problem permanently. The task is to create a `Dockerfile` that starts from an Ubuntu base, uses the package manager to install `curl` during the build phase, copies a local bash script into the image, and sets that script to execute automatically when the container runs.

---

## Hints

* **Base Image:** Start with the exact Ubuntu version specified in the instructions using the `FROM` command.
* **Dependencies:** Use the `RUN` command to execute the `apt-get update` and `apt-get install` commands. Remember to include flags like `-y` so the installation doesn't pause waiting for user confirmation during the automated build.
* **Adding Files:** Use the `COPY` command to move your local `script.sh` into the container's filesystem.
* **Execution:** Ensure the script has execution permissions (either locally before copying, or via a `RUN chmod` command in the Dockerfile). Set the final execution instruction using `CMD`.

## Verification

### Expected Result

Building the image should complete successfully, executing the apt-get installation steps as layers. When running the resulting image interactively, it should immediately prompt for a website, and providing one should successfully fetch the HTTP response using the newly installed `curl` command, without any "not found" errors.

---

## Challenges Faced

* Dealing with the interactive prompt of `apt-get install` halting the build process because it was waiting for a "Yes" confirmation. (Solved using the `-y` flag).
* "Permission denied" errors when the container tried to execute the script. (Solved by ensuring the script was executable).

---

## Lessons Learned

* **Infrastructure as Code:** A `Dockerfile` replaces manual, error-prone setup steps. Anyone can take this `Dockerfile` and build an environment identical to mine.
* **Build vs Run:** The distinction between `RUN` (which happens once during image creation) and `CMD` (which happens every time a container is spun up from the image) is critical for structuring Dockerfiles correctly.
