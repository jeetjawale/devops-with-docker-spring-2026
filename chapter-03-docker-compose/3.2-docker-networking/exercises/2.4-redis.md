# Exercise 2.4 - Redis

## Objective

Enhance a multi-container application by introducing a caching layer (Redis) that the backend communicates with via Docker Compose's internal network.

---

## Concepts Practiced

- Internal Docker Networking
- DNS Resolution by Service Name
- Caching (Redis)
- Service Dependencies and Restart Policies

---

## Exercise Summary

This exercise expands on the previous frontend-backend setup. A slow API endpoint exists in the backend that can be optimized using a Redis cache. The task is to add a Redis service to the `compose.yaml` file and configure the backend to connect to it. The focus is on ensuring the connection happens purely over the internal Docker network without exposing the Redis database to the host machine.

---

## Hints

- Check the backend's README to find which environment variable it expects for the Redis connection string.
- In a default Compose setup, a container's hostname is its service name. Use this service name in the backend's environment variables.
- **Security Check:** You do not need to use the `ports` key for the Redis service. The backend can access it internally on the default Redis port (6379).
- If the backend crashes because it starts up faster than Redis, consider using the `restart: unless-stopped` policy on the backend service to allow it to recover automatically.

---

## Commands Used

```bash
# Bring up the expanded application stack
docker compose up
```

---

## Verification

### Expected Result

When the stack is running, clicking the relevant button in the frontend (which hits the `/ping?redis=true` endpoint) should turn green, verifying that the backend successfully retrieved data from the newly configured Redis cache.

---

## Challenges Faced

- Configuring the backend to point to the correct internal hostname.
- Handling race conditions where the backend tries to connect to the database before the database is fully initialized.

---

## Lessons Learned

- Docker Compose automatically provides internal DNS resolution based on service names, eliminating the need to hardcode IP addresses.
- Internal databases and caches should remain isolated. They do not require `ports` mappings unless you need to access them directly from your host machine for debugging.
