# Exercise 2.5 - Scale

## Objective

Learn how to horizontally scale compute containers using Docker Compose to distribute load behind a load balancer.

---

## Concepts Practiced

- Horizontal Scaling
- Load Balancing
- Avoiding Port Clashes

---

## Exercise Summary

In this exercise, you clone a repository containing a deliberately struggling application placed behind a load balancer. The application cannot process requests fast enough to turn a UI button green. The task is to scale the compute instances horizontally using the `docker compose` CLI so that the load balancer can distribute traffic across multiple workers, thereby increasing throughput.

---

## Hints

- Review the provided `docker-compose.yml` to identify the name of the service doing the computation (the one that needs scaling).
- Note how the compute service is defined: it does *not* bind to a static host port, which is what makes scaling possible without port conflicts.
- Use the `--scale` flag when bringing up the compose environment.

---

## Commands Used

```bash
# Bring up the stack and scale a specific service
docker compose up --scale <service_name>=<number_of_instances>
```

---

## Verification

### Expected Result

After scaling the compute containers sufficiently (e.g., to 3 or more instances), accessing the application at `http://localhost:3000` and interacting with it should result in a green button, confirming the load is being successfully distributed and processed.

---

## Challenges Faced

- Determining the correct CLI syntax to scale a specific service.
- Identifying which specific service in the architecture required scaling.

---

## Lessons Learned

- Horizontal scaling locally is extremely straightforward with Docker Compose via the `--scale` flag.
- For scaling to work on a single host machine, the scaled containers cannot bind to a fixed host port (`host:container`). They must either bind to a dynamic port or operate entirely within the internal network behind a proxy/load balancer.
