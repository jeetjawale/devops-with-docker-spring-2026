# Exercise 2.1 - Simple service writing to log

## Objective

Transition from running a container with `docker run` to using Docker Compose, specifically focusing on translating volume bind mounts into a declarative compose configuration.

---

## Concepts Practiced

- Docker Compose Configuration
- Volume Bind Mounts
- Container Logging

---

## Exercise Summary

The goal of this exercise is to take the `simple-web-service` image used previously and run it via Docker Compose. The service writes logs to `/usr/src/app/text.log` by default. The challenge is to configure a volume mount within the `docker-compose.yaml` file so that this log file is synced to the host filesystem, verifying that the service runs and logs can be accessed locally.

---

## Hints

- Review the syntax for defining volumes within a service in a Docker Compose file.
- Remember the mapping order: `[host-path]:[container-path]`.
- You can map a directory or a specific file, but ensure the host target exists or is defined correctly relative to the `compose.yaml` file.

---

## Commands Used

```bash
# Start the service defined in compose.yaml
docker compose up

# Start the service in the background
docker compose up -d

# Verify logs if running detached
docker compose logs
```

---

## Verification

### Expected Result

When you run `docker compose up`, the service starts. A log file should appear in the configured host directory, containing the output generated by the containerized service.

---

## Challenges Faced

- Translating the `docker run -v` command-line argument into the equivalent YAML structure under the `volumes` key.
- Ensuring the correct path inside the container (`/usr/src/app/text.log`) is targeted.

---

## Lessons Learned

- Defining volume mounts in a `compose.yaml` file is much cleaner and less prone to typos compared to typing long `docker run` commands.
- Docker Compose handles the underlying Docker API calls to create and bind the volumes transparently.
