# Exercise 2.11 - Your dev env

## Objective

Apply the concepts of Docker Compose to a real-world, personal project to standardize and simplify its local development environment.

---

## Concepts Practiced

- Containerizing Development Environments
- Translating Local Workflows to `compose.yaml`
- Bind Mounts and Hot Reloading
- Database/Dependency Isolation

---

## Exercise Summary

This open-ended exercise requires taking an existing personal project and introducing Docker to its development workflow. This can be approached incrementally: starting by simply containerizing heavy external dependencies (like a local database or caching layer) using Docker Compose, or going all the way to creating a fully containerized environment where the application code itself runs inside Docker, utilizing bind mounts to enable instant hot-reloading during active development.

---

## Hints

- **Start Small:** If your project requires a database (e.g., PostgreSQL, MongoDB, Redis), write a `compose.yaml` that spins up just those services. This instantly frees you from managing local database installations.
- **Hot Reloading:** If you choose to containerize your application code, ensure you use a bind mount (`.:/app`) to map your local source code into the container so you don't have to rebuild the image every time you save a file.
- **Protect Dependencies:** Remember the "volume trick" learned in this chapter. If your language uses a local dependency folder (like `node_modules` for JS or `.venv` for Python), create a specific named volume for that folder (e.g., `node_modules:/app/node_modules`) to prevent your host machine from corrupting the container's native binaries.
- Review your project's `README.md` or setup script. Every manual "install this" or "run that" step should ideally be translated into the `Dockerfile` or `compose.yaml`.

---

## Commands Used

```bash
# Build the images and start the development environment
docker compose up --build

# Tear down the environment when finished
docker compose down
```

---

## Verification

### Expected Result

You should be able to clone your project onto a fresh machine (that only has Docker installed) and run `docker compose up`. The required databases should start, the application should boot, and (if fully containerized) changing a line of code in your editor should instantly trigger a reload inside the running container.

---

## Challenges Faced

- Translating complex, manual local setup instructions into a declarative `Dockerfile`.
- Handling language-specific environment quirks (e.g., ensuring development-specific environment variables are loaded correctly, or managing package manager caching).
- Ensuring file-watching tools (like `nodemon` or Webpack) correctly detect filesystem events across the Docker boundary.

---

## Lessons Learned

- Setting up a containerized development environment requires a deeper architectural understanding of how the application runs, but it pays off massively by completely eliminating the "it works on my machine" problem.
- A well-crafted `compose.yaml` serves as executable, self-documenting infrastructure for your project.
