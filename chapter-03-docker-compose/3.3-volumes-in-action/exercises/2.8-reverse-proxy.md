# Exercise 2.8 - Reverse proxy

## Objective

Set up Nginx as a reverse proxy to act as the single entry point for both the frontend and backend services.

---

## Concepts Practiced

- Reverse Proxies (Nginx)
- Traffic Routing
- Configuration Bind Mounting

---

## Exercise Summary

This exercise introduces a reverse proxy to manage incoming traffic. Instead of accessing the frontend and backend on different ports, all traffic will go to `http://localhost:8000`. An Nginx container is added to the stack, configured to route traffic based on the URL path: requests starting with `/api/` are forwarded to the backend, while all other requests (`/`) go to the frontend.

---

## Hints

- You need an `nginx.conf` file on your host machine containing the routing rules provided in the exercise description.
- Use a bind mount to mount this local `nginx.conf` into the Nginx container at `/etc/nginx/nginx.conf`.
- Inside `nginx.conf`, the `proxy_pass` directives should use the internal service names (e.g., `http://frontend:port` and `http://backend:port`).
- Ensure you map port 8000 on your host to port 80 on the Nginx container.

---

## Commands Used

```bash
# Start the stack with the new proxy
docker compose up
```

---

## Verification

### Expected Result

Navigating to `http://localhost:8000` should display the frontend application. Navigating directly to `http://localhost:8000/api/ping` should return the backend's response (e.g., `pong`). (Note: Some frontend buttons might be broken; this is expected and will be fixed in the next exercise).

---

## Challenges Faced

- Writing the correct `proxy_pass` URLs using internal Docker DNS.
- Ensuring the syntax inside `nginx.conf` is perfectly valid. If the Nginx container exits immediately, checking `docker compose logs nginx` is vital.

---

## Lessons Learned

- A reverse proxy drastically simplifies the client-side architecture by unifying everything under a single domain/port.
- Nginx can seamlessly resolve Docker Compose service names to internal IP addresses.
