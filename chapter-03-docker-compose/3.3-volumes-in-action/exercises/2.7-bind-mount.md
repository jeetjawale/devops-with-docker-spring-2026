# Exercise 2.7 - Bind mount

## Objective

Transition the PostgreSQL database storage from an anonymous volume to a local bind mount to ensure explicit data persistence.

---

## Concepts Practiced

- Bind Mounts
- Data Persistence Validation
- Volume Mapping Syntax

---

## Exercise Summary

While the previous exercise used an anonymous volume created implicitly by the Postgres image, this exercise tasks you with explicitly configuring a bind mount. You must map a local directory (e.g., `./database`) to the internal directory where Postgres stores its data. You then verify persistence by creating data, destroying the containers, and bringing them back up to ensure the data survived. Finally, you verify that manually deleting the bind mount directory destroys the data.

---

## Hints

- Check the Postgres image documentation to find exactly which internal container path needs to be persisted (typically `/var/lib/postgresql/data`).
- Update the `volumes` section of your Postgres service in `compose.yaml` to map a relative host path to the internal path.
- Remember the mapping syntax is `[host-path]:[container-path]`.

---

## Commands Used

```bash
# Bring the stack up
docker compose up -d

# Bring the stack down (removes containers, preserves bind mount data)
docker compose down

# Delete the local bind mount directory to reset data
rm -rf ./database
```

---

## Verification

### Expected Result

1. Start the stack and save messages via the frontend.
2. Run `docker compose down`, then `docker compose up` again.
3. Refresh the browser; the messages should still be there because the data is saved in `./database` on your host.
4. Run `docker compose down`, delete `./database`, and run `docker compose up` again. The messages should be gone.

---

## Challenges Faced

- Finding the exact internal path where Postgres stores its data.
- Handling potential file permission issues on the host machine when the container creates the `./database` folder.

---

## Lessons Learned

- Bind mounts make it very easy to locate and backup data directly from the host filesystem.
- Bringing down a compose stack does not destroy bind-mounted data, ensuring data safety across container recreations.
