# Exercise 2.10 - Close the ports

## Objective

Secure the application by removing unnecessary port exposures, ensuring that all traffic is forced exclusively through the reverse proxy.

---

## Concepts Practiced

- Docker Network Security
- Port Publishing vs Internal Exposure
- Network Auditing (nmap)

---

## Exercise Summary

Even though the Nginx proxy is functioning, the frontend and backend services likely still have their `ports` mapped in `compose.yaml`. This means they are still directly accessible from the host network, bypassing the proxy. This exercise requires removing these redundant port mappings and validating the closed ports using a port scanner (`nmap`).

---

## Hints

- Review your `compose.yaml` file. Look for `ports:` declarations under every service except `nginx`.
- Remember that `ports:` maps a container port to the host. If you delete the `ports:` block, the container can still be reached by other containers on the same Docker network (like Nginx) on its internal port.
- Use the provided `nmap` Docker command to scan localhost before and after making your changes.

---

## Commands Used

```bash
# Scan localhost for open ports
docker run -it --rm --network host networkstatic/nmap localhost

# Apply compose changes
docker compose up -d
```

---

## Verification

### Expected Result

1. Before changes: Running the `nmap` command will show ports like 5001 (frontend) and 8080 (backend) as open/filtered.
2. After removing `ports:` mappings and restarting: Running the `nmap` command will only show port 8000 (Nginx proxy) as open. Ports 5001 and 8080 should not appear in the scan.

---

## Challenges Faced

- Trusting that the internal network will still work without explicit port mappings. It can be counterintuitive initially that Nginx can reach a port that isn't explicitly "published".

---

## Lessons Learned

- Publishing ports (`ports:`) is strictly for granting access from *outside* the Docker network (the host machine).
- By closing internal ports, you drastically reduce the attack surface of your application, ensuring all traffic must pass through the proxy where security rules, rate limiting, and logging can be centrally enforced.
