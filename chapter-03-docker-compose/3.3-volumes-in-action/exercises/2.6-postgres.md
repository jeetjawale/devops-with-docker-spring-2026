# Exercise 2.6 - Postgres

## Objective

Integrate a PostgreSQL database into the existing multi-container backend/frontend stack using Docker Compose.

---

## Concepts Practiced

- Database Container Integration
- Anonymous Volumes
- Service Environment Variables

---

## Exercise Summary

This exercise requires adding a PostgreSQL service to the `compose.yaml` file so the backend can save and retrieve messages. At this stage, no explicit volume configuration is required because the official Postgres image defines an internal volume, resulting in an anonymous volume being created by default. The key task is to configure the correct environment variables for both the database and the backend so they can communicate over the internal Docker network.

---

## Hints

- Review the official Postgres image documentation on Docker Hub to find the mandatory environment variable required to start the container (typically related to setting a password).
- Check the backend application's README to find which environment variable it expects for the database connection string.
- If the backend fails to connect immediately because the database is still initializing, setting `restart: unless-stopped` on the backend service can help it recover.
- If you encounter leftover data issues while iterating, remember to clean up old anonymous volumes using `docker volume prune`.

---

## Commands Used

```bash
# Clean up leftover anonymous volumes from previous runs
docker compose down -v
# OR manually prune
docker volume prune

# Start the stack
docker compose up
```

---

## Verification

### Expected Result

Running `docker compose up` should successfully boot the database, backend, and frontend. Navigating to the frontend and interacting with the database form should allow you to save messages, confirming the backend is successfully writing to Postgres.

---

## Challenges Faced

- Dealing with leftover anonymous volumes causing conflicts or authentication errors across restarts.
- Matching the environment variables expected by the backend with the credentials defined in the Postgres service.

---

## Lessons Learned

- Official images often declare `VOLUME` internally. If you don't map them explicitly, Docker creates anonymous volumes that can persist between runs and cause unexpected state if not pruned.
- Internal service names (e.g., `db`) act as hostnames inside the Compose network.
