# Exercise 3.4: Building images from inside a container

## Objective

Learn how to use "Docker-out-of-Docker" concepts by binding the host's Docker socket to a container, allowing a containerized script to execute native Docker commands like build and push.

---

## Concepts Practiced

- Docker-in-Docker (DinD) / Docker-out-of-Docker (DooD)
- Docker Socket Binding (`/var/run/docker.sock`)
- Containerized Build Scripts
- Environment Variable Secrets Management
- Docker `ENTRYPOINT` configurations

---

## Exercise Summary

This exercise took the bash build script from Exercise 3.3 and fully "Dockerized" it. Instead of running the script directly on the host machine, it was packaged into a Docker image based on the lightweight `docker:cli` Alpine image. The container was configured to take over execution using an `ENTRYPOINT`. When running the container, the host machine's Docker socket was mounted as a volume. This allowed the Dockerized script to issue standard `docker build` and `docker push` commands that actually executed on the host's Docker daemon. Since the script is running isolated inside a container, Docker Hub credentials were passed in securely as runtime environment variables rather than relying on the host's local Docker login state.

---

## Hints

- Start your Dockerfile with `FROM docker:cli` (or a similar Alpine image that includes the Docker binary). Since you need to clone repositories inside the script, don't forget to install `git` within your Dockerfile via the Alpine package manager (`apk add --no-cache git`).
- Use the `ENTRYPOINT` instruction in your Dockerfile to ensure your custom script executes immediately when the container starts. Any arguments passed to `docker run` at the end of the command will automatically be appended as arguments to your `ENTRYPOINT` script.
- The internal script needs to log in to Docker Hub itself. You can do this securely by piping the password directly into the login command to avoid warnings about insecure passwords: `echo "$DOCKER_PWD" | docker login -u "$DOCKER_USER" --password-stdin`.
- To make your bash script extremely robust, look into using the `trap` command. Setting a `trap cleanup EXIT` ensures that your temporary directories are deleted even if the script crashes unexpectedly or is terminated early by the user.

---

## Commands Used

```bash
# Example command to run the containerized builder
docker run -e DOCKER_USER=myusername \
  -e DOCKER_PWD=mypassword \
  -v /var/run/docker.sock:/var/run/docker.sock \
  my-builder-image mygithubuser/myrepo mydockeruser/myimage
```

---

## Verification

### Expected Result

When executing the `docker run` command with the appropriate environment variables and the Docker socket mounted, the container should spin up and immediately execute the internal build script. The terminal should output the `git clone` progress, a successful Docker Hub login message, the `docker build` layers, and finally the `docker push` logs. You can verify success by checking your remote Docker Hub repository for the newly uploaded image.

---

## Challenges Faced

- Handling authentication within an isolated container without exposing passwords in the command line history (solved by utilizing environment variables).
- Understanding how mounting `/var/run/docker.sock` breaks the typical container isolation boundary to give the container root-level access to the host's Docker engine.

---

## Lessons Learned

- Mounting the Docker socket is an incredibly powerful feature that essentially allows containers to control the host system's container engine. This is exactly how tools like Watchtower, Portainer, and standard Jenkins/GitLab CI runners operate.
- Because mounting the Docker socket grants full root privileges over the host's Docker daemon, it introduces massive security implications and should only be done with deeply trusted images.
- Packaging scripts inside containers eliminates the "it works on my machine" problem, as the user no longer needs to have Git, Bash, or specific CLI tools pre-installed on their host machine to run complex build pipelines.

## Completion Notes

Seeing a container build and push another container is a wild concept at first, but it perfectly explains how major cloud CI/CD providers work under the hood. It really solidifies the idea that Docker is just a background service running on a host, and the Docker CLI is just a client API that can interact with it from anywhere—even from inside another container!
