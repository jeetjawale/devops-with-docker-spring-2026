# Exercise 3.3: Scripting magic

## Objective

Build a custom deployment script that automates the entire process of fetching source code from GitHub, building a Docker image, and publishing it to Docker Hub.

---

## Concepts Practiced

- Shell Scripting (Bash)
- Automation and CI/CD Internals
- Docker CLI Workflows
- Git Integration

---

## Exercise Summary

This exercise involved creating a custom executable script that acts as a simple, local CI/CD runner. The script accepts exactly two arguments: a target GitHub repository path and a destination Docker Hub repository name. When executed, it automatically clones the specified GitHub repository into a fresh temporary workspace, uses the repository's `Dockerfile` to build a new image, tags it with the provided Docker Hub name, and pushes it to the remote registry. Finally, it cleans up the temporary files. This exercise perfectly simulates the exact sequential steps a cloud CI tool like GitHub Actions performs under the hood.

---

## Hints

- Bash is usually the most straightforward choice for this kind of automation because it handles standard CLI tools (like `git` and `docker`) very naturally without needing external libraries.
- Consider what happens if the script fails halfway through. Using `set -e` at the top of a Bash script ensures that the script instantly aborts if any single command (like `git clone` or `docker build`) fails, preventing it from attempting to push a broken or empty image.
- Since you are cloning remote repositories, it is best to isolate the build environment. Using the `mktemp -d` command in Bash securely creates a temporary directory that you can clone code into, ensuring you don't pollute your local file system. Always remember to recursively delete this temporary directory at the end of the script!
- You must ensure you are already authenticated with the Docker CLI locally (via `docker login`) before running the script; otherwise, the final `docker push` step will fail due to a lack of authorization.

## Commands Used

```bash
# Making the script executable
chmod +x builder.sh

# Example execution of the script
./builder.sh mluukkai/express_app my_dockerhub_user/testing_repo
```

## Verification

### Expected Result

Running the script with valid repository paths successfully pulls the latest code from GitHub. The terminal output should show Git cloning the files, followed by the familiar step-by-step Docker build logs, and concluding with the Docker push logs. Checking your Docker Hub account afterward via the web browser should reveal the newly pushed image under the specified repository name.

---

## Challenges Faced

- Handling script arguments securely and verifying that the user provided exactly two parameters before executing any logic.
- Ensuring the local environment stays clean by properly managing temporary directories, even if the script were to fail.

---

## Lessons Learned

- Complex CI/CD platforms like GitHub Actions or Jenkins are ultimately just executing automated shell scripts similar to the one built in this exercise.
- Shell scripting is a vital "glue" in the DevOps ecosystem, allowing you to seamlessly connect disparate tools (like `git` and `docker`) into unified, powerful pipelines.
- Leveraging temporary, stateless directories (`mktemp`) is a best practice for automation tasks to prevent side-effects on the host machine.
