# Exercise 3.1: Your pipeline

## Objective

Learn how to build an automated continuous integration and continuous deployment (CI/CD) pipeline for a containerized application using GitHub Actions and Watchtower.

---

## Concepts Practiced

- CI/CD Pipelines
- GitHub Actions Workflows
- Automated Image Building and Pushing
- Watchtower (Automated Container Updates)
- Secrets Management (Docker Hub Credentials)

---

## Exercise Summary

The exercise involved taking a provided Express.js application, pushing it to a GitHub repository, and setting up a complete deployment pipeline. A GitHub Actions workflow was configured to trigger whenever new code was pushed to the `main` branch. This workflow securely authenticated with Docker Hub using secrets, then automatically built and pushed the new Docker image. Finally, Watchtower was set up alongside the application container to actively poll Docker Hub and automatically deploy the updated container as soon as the pipeline finished publishing the new image.

---

## Hints

- When configuring the GitHub Actions YAML, always check the official documentation for the actions (`checkout`, `login-action`, `build-push-action`, `setup-buildx-action`) to ensure you are using their latest major versions (e.g., `v4` or `v5`) instead of outdated ones.
- Never hardcode your Docker Hub credentials. Use GitHub repository secrets to securely pass your username and access token into the workflow runner.
- If you run the built image locally on an ARM-based machine (like an Apple M1/M2 Mac), you might encounter an `exec format error`. This happens because standard GitHub runners build for `amd64`. You can fix this by either running the local container with Rosetta translation or updating the GitHub action to build multi-platform images using `setup-qemu-action`.
- Ensure Watchtower has access to the host's Docker socket (`/var/run/docker.sock`) so it has the permissions required to detect, stop, and restart other containers.

## Commands Used

```bash
# Standard workflow to trigger the CI/CD pipeline
git commit -am "Update frontend text"
git push origin main

# Monitoring the local deployment via Watchtower
docker compose up -d
docker compose logs -f watchtower
```

---

## Verification

### Expected Result

Pushing a code change to the repository automatically spins up a GitHub Actions runner. Once the workflow completes, the new image appears on Docker Hub. Within a few minutes (depending on the polling interval), the local Watchtower instance detects the update, pulls the new image, and seamlessly swaps out the running container. Refreshing the browser should immediately display the updated code changes.

---

## Challenges Faced

- Dealing with potential runner architecture mismatch issues (building on `linux/amd64` but running locally on Apple Silicon).
- Ensuring the syntax inside the GitHub Actions YAML was strictly formatted.

---

## Lessons Learned

- GitHub Actions provides a very tight, seamless integration with Docker Hub for automating the build process.
- Watchtower is an incredibly lightweight and effective solution for updating containers in small-scale environments without the heavy overhead of Kubernetes.
- Using dedicated Personal Access Tokens (PATs) for Docker Hub authentication is a much safer practice than using standard account passwords in CI/CD pipelines.
