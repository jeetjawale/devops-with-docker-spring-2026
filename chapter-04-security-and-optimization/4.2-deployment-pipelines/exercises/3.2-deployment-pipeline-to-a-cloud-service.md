# Exercise 3.2: Deployment pipeline to a cloud service

## Objective

Enhance a previously containerized application by establishing an automated continuous deployment pipeline to a cloud Platform as a Service (PaaS) using GitHub Actions.

---

## Concepts Practiced

- Cloud Deployment (PaaS)
- GitHub Actions Integration
- Third-party CI/CD Actions
- Automated Cloud Provisioning
- Application Delivery

---

## Exercise Summary

This exercise involved taking a containerized application deployed in a previous exercise (Exercise 1.16) and fully automating its release cycle. Instead of manually pushing images or triggering deployments from a local machine, a GitHub Actions workflow was configured. This workflow listens for pushes to the main branch and utilizes a pre-built GitHub Action designed for the target cloud provider. The pipeline automatically authenticates with the cloud service and seamlessly pushes the new Docker image into production without human intervention.

---

## Hints

- Look for official or highly-rated community GitHub Actions for your specific cloud provider (e.g., Render, Fly.io, Heroku, or AWS). Using these pre-built actions from the GitHub Marketplace saves you from writing custom bash scripts to interact with cloud APIs.
- Make sure you inject your cloud provider's API tokens or deploy keys as GitHub Secrets (`Settings > Secrets and variables > Actions`) rather than committing them directly in your `.yaml` workflow file.
- Depending on the cloud provider, you might only need to trigger a webhook, or you might need to build the Docker image natively in GitHub Actions and push it to the provider's internal container registry. Always check the provider's specific documentation for CI/CD.

---

## Commands Used

```bash
# Typical local Git workflow to trigger the cloud deployment
git commit -m "Update app for automated cloud deployment"
git push origin main
```

---

## Verification

### Expected Result

When code is pushed to the repository's `main` branch, the GitHub Action workflow successfully executes. Once the pipeline completes, visiting the cloud application's public URL (which is provided in the repository's README) immediately reflects the latest code changes. No manual deployment commands (like `docker push` or proprietary cloud CLI commands) are necessary.

---

## Challenges Faced

- Configuring the correct authentication method and required environment variables for the specific cloud provider inside the GitHub Actions runner.
- Ensuring the container listens to the dynamic `$PORT` variable injected by the cloud provider, rather than a hardcoded port used during local testing.

---

## Lessons Learned

- Community-created GitHub Actions drastically simplify integrating with external APIs and major cloud providers.
- Removing manual deployment steps reduces human error and enforces a "single source of truth" (the Git repository) for what is running in production.
- Continuous deployment pipelines are completely independent of the application code itself. The exact same containerized application can be shifted from local deployment (via Watchtower) to a cloud provider simply by swapping out the CI/CD pipeline steps.
