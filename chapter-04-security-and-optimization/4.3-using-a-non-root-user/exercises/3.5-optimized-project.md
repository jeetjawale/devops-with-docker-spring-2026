# Exercise 3.5: Optimized project (mandatory)

## Objective

Secure Docker containers by ensuring the primary application processes do not run as the root user.

---

## Concepts Practiced

- Docker Security Best Practices
- Principle of Least Privilege
- The `USER` Dockerfile Instruction
- Linux User and Permission Management (`useradd`, `chown`)

---

## Exercise Summary

This exercise focused on addressing a major security vulnerability in the example frontend and backend applications from Chapter 1. By default, processes inside a Docker container run as the `root` user. This means if an attacker manages to compromise the application running inside the container (e.g., via a remote code execution vulnerability), they immediately gain root-level access to the container environment. To mitigate this risk, both Dockerfiles were updated to create a dedicated, unprivileged system user. Ownership of the application workspace was transferred to this new user, and the Dockerfile was instructed to switch away from `root` to this user right before executing the final web service commands. 

---

## Hints

- You need to execute the user creation commands (`useradd` or `adduser`, depending on whether your base image uses Debian/Ubuntu or Alpine) *before* you switch the Dockerfile context with the `USER` instruction.
- The newly created non-root user will not have permission to read or modify files in directories that were previously created by root (such as the default `WORKDIR`). You must explicitly grant ownership of the application directory to the new user using the Linux `chown` command.
- The `USER` instruction changes the active user for all subsequent `RUN`, `CMD`, and `ENTRYPOINT` instructions. Place it towards the very end of your Dockerfile, right before the `EXPOSE` and `CMD` lines. This ensures that earlier software installation steps (like `apt-get install` or global `npm install`), which require root privileges, can still complete successfully.

---

## Commands Used

```bash
# Example Linux commands used inside the Dockerfile RUN instruction
useradd -m appuser
chown -R appuser:appuser /usr/src/app
```

---

## Verification

### Expected Result

After building the updated images, starting the containers should behave exactly as before, and the frontend and backend should still communicate seamlessly over the local network. However, if you execute an interactive shell inside the running container (`docker exec -it <container_name> sh`) and type `whoami`, the terminal should return your custom username (e.g., `appuser`) instead of `root`.

---

## Challenges Faced

- Encountering unexpected "permission denied" or "EACCES" errors when the application attempted to start. This typically happens if the `chown` command is forgotten or applied to the wrong path.
- Remembering that different Linux distributions use different user management commands. For instance, Ubuntu uses `useradd -m`, whereas Alpine Linux relies on `adduser -D`.

---

## Lessons Learned

- Running containers as root is a massive, easily preventable security risk, especially for public web-facing services.
- Applying the Principle of Least Privilege in Docker is conceptually straightforward but requires a solid understanding of standard Linux file permissions and user ownership.
- The order of operations in a Dockerfile is absolutely critical; changing to a non-root user too early in the build process will immediately break package managers and global installations.

## Completion Notes

Implementing non-root users is one of the quickest and most impactful "wins" in container security. It is widely considered a mandatory step for any container that is going to be exposed to the open internet in a production environment.
