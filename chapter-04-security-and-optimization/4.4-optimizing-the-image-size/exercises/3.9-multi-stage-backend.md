# Exercise 3.9: Multi-stage backend

## Objective

Leverage the power of Go's static compilation and Docker's `scratch` image to create an incredibly minimal, ultra-secure production container for the backend application.

---

## Concepts Practiced

- Docker `scratch` image
- Static compilation in Go (`CGO_ENABLED=0`)
- Advanced Multi-stage copying (`/etc/passwd`, CA certificates)
- Ultimate image size optimization

---

## Exercise Summary

This exercise took multi-stage builds to their absolute logical extreme. Because Go can compile code into a completely self-contained, statically linked binary, it doesn't actually need an operating system (like Ubuntu or Alpine) to run. 

The first stage of the build used a standard `golang:alpine` image to install CA certificates, create a non-root user (`appuser`), and compile the Go code with CGO disabled. 

The second stage used `FROM scratch`, which is literally an empty file system containing exactly 0 bytes. From the first stage, the compiled binary, the SSL certificates, and the Linux user file (`/etc/passwd`) were copied into this empty void. The resulting final image contained only the application binary and the absolute bare minimum configuration files required to run it, easily beating the 35 MB requirement (typically coming in under 20 MB).

---

## Hints

- By default, Go binaries might dynamically link to C libraries (like `glibc` or `musl`). An empty `scratch` container doesn't have these libraries, so the app will crash instantly. You must force Go to build a completely static binary by prefixing your build command with environment variables: `CGO_ENABLED=0 GOOS=linux go build -o server .`.
- If your backend needs to make HTTPS requests to the internet, it needs CA certificates to verify SSL connections. Since `scratch` is completely empty, it doesn't have any certificates. You must install them in your build stage (e.g., `apk add --no-cache ca-certificates`) and then manually `COPY` them to the scratch container (usually into `/etc/ssl/certs/ca-certificates.crt`).
- You can't run `adduser` or `useradd` in a `scratch` container because there is no shell (`/bin/sh`) or user management software! The trick is to create the user in the build stage, and then `COPY --from=build-stage /etc/passwd /etc/passwd` into the `scratch` container. Docker will then be able to recognize your `USER` directive.

---

## Commands Used

```bash
# Compiling a completely static Go binary for Linux
CGO_ENABLED=0 GOOS=linux go build -o server .
```

## Verification

### Expected Result

When you run `docker images`, the backend image size should be astonishingly small—essentially just the size of the compiled binary itself. Starting the backend container should work seamlessly, it should still run as the non-root user, and it should still be able to securely communicate with external APIs (thanks to the copied certificates).

---

## Challenges Faced

- **"executable file not found in $PATH"**: This cryptic error occurs if you try to run a dynamically linked Go binary in a scratch container. Disabling CGO (`CGO_ENABLED=0`) fixes it.
- **"x509: certificate signed by unknown authority"**: This error occurs if the application attempts an HTTPS request but the `ca-certificates` were not copied into the scratch container.
- **"unable to find user"**: This happens if you set `USER appuser` in the scratch stage, but forgot to copy over the `/etc/passwd` file from the build stage.

---

## Lessons Learned

- The `scratch` image is the gold standard for deploying compiled languages (Go, Rust, C++). It provides unparalleled security because an attacker literally has zero tools (no `bash`, no `curl`, no `ls`) if they manage to break into the container.
- When working with `scratch`, you suddenly realize how many things you take for granted in a normal OS (like SSL certificates, timezone data, and user management files). Everything you need must be explicitly copied.
