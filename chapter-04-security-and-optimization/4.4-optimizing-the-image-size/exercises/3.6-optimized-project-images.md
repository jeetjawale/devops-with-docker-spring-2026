# Exercise 3.6: Optimized project images

## Objective

Reduce the overall size of Docker images by optimizing Dockerfile instructions, chaining commands, and aggressively cleaning up unused files and build dependencies.

---

## Concepts Practiced

- Docker Layer Optimization
- Package Manager Caching (`apt-get clean`)
- Stripping Development Dependencies
- Command Chaining (`&& \`)

---

## Exercise Summary

This exercise focused on significantly trimming down the bloated image sizes of the frontend and backend applications from Chapter 1 without changing their underlying operating system base images (Ubuntu). Initially, these images were over 1.1 GB and 1.5 GB, respectively. By chaining all setup, installation, and configuration steps into a single massive `RUN` instruction, the creation of unnecessary intermediate layers was avoided. 

Furthermore, aggressive cleanup strategies were implemented. For the frontend, once the static files were built, the raw `src` code and the massive `node_modules` folder were completely deleted. For the backend, the Go compiler itself (`golang`) was uninstalled via the package manager immediately after the binary was compiled. Additionally, APT caches and Go build caches were wiped. These optimizations cut the final image sizes down by more than half (to ~591 MB and ~251 MB).

---

## Hints

- Deleting files in a separate `RUN` instruction does **not** save space, as the deleted files still exist in the underlying layer. All cleanup commands must occur in the exact same `RUN` step that created the files. Use `&& \` to chain them.
- When you use `apt-get install`, it downloads compressed repository package lists. You can safely remove these after installation using `rm -rf /var/lib/apt/lists/*` to save 20-40 MB.
- Think about what your application *actually needs* to run in production. For a compiled React frontend, it only needs a web server (`serve`) and the `build/` folder. Everything else (`src`, `public`, `package.json`, `node_modules`) can be deleted.
- If you use a package manager to install a compiler (like `golang` or `curl`), you can use `apt-get purge -y --auto-remove <package>` to completely uninstall it and its unneeded dependencies after the compilation is finished.

---

## Commands Used

```bash
# General syntax for purging packages and cleaning up APT in Ubuntu
apt-get purge -y --auto-remove package_name
rm -rf /var/lib/apt/lists/*

# Viewing the difference in image sizes
docker images
```

---

## Verification

### Expected Result

Running `docker images` before and after the Dockerfile changes should show a dramatic drop in size (often a 50% to 80% reduction). Despite the massive removal of files, building and running the containers via `docker compose up` should yield perfectly functioning frontend and backend applications that communicate normally.

---

## Challenges Faced

- Ensuring the strict syntax of command chaining was correct. A missing `\` or `&&` can easily break the entire Docker build process.
- Finding exactly which directories the Go compiler uses for caching (`/root/.cache` and `/go`) so they could be manually deleted.

---

## Lessons Learned

- Source code and build tools (like compilers) have absolutely no place in a production container. They bloat the image and increase the security attack surface.
- The default `ubuntu` base image is very heavy. While optimizing `RUN` commands is a great practice, it highlights that using an inherently smaller base image (like Alpine) or multi-stage builds is ultimately the best long-term strategy for minimizing size.

## Completion Notes

Manually uninstalling compilers and deleting source code directories inside a single 15-line `RUN` command feels very "hacky" and is incredibly tedious to maintain. However, manually doing this exercise perfectly illustrates *why* Docker eventually introduced Multi-stage builds, which automate this exact separation of concerns much more cleanly.
