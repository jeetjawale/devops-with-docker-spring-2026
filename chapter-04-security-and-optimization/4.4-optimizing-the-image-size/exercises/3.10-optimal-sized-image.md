# Exercise 3.10: Optimal sized image

## Objective

Apply all the security and size optimization techniques learned throughout this chapter to a completely independent, standalone project of your choosing.

---

## Concepts Practiced

- Comprehensive Dockerfile Optimization
- Applying Security Best Practices (Non-root user)
- Alpine Base Images Migration
- Directory Permission Management (`chown`)

---

## Exercise Summary

For this final exercise of the section, the standalone `express-app` project (a basic Node.js Express server) was selected for optimization. 

The original Dockerfile was extremely basic: it used the heavy `node:16` image, copied the files to the root of the container without a dedicated `WORKDIR`, installed dependencies, and ran as the default `root` user. 

The optimization process overhauled this entirely. First, the base image was swapped to `node:16-alpine` to drastically reduce the initial image footprint (saving hundreds of megabytes). Second, a dedicated `WORKDIR` (`/usr/src/app`) was established for cleaner file organization. Finally, the security vulnerability was addressed by chaining a `RUN` command that created a dedicated `appuser` (using Alpine's `adduser -D` syntax), granted that user ownership of the working directory, and safely dropped root privileges via the `USER appuser` directive before starting the Node server.

---

## Hints

- You can pick *any* project you have locally. The easiest targets are the simple backend projects you wrote during Part 1.
- You do not need a complex multi-stage build if the project is interpreted (like Python or Node.js). Simply switching to an Alpine variant is usually enough to achieve a massive size reduction.
- Don't forget that if you change your base image to Alpine, your user creation commands must change from Ubuntu's syntax (`useradd -m`) to Alpine's syntax (`adduser -D`).
- Always verify that your application still runs after optimization! Sometimes Alpine images are missing native C++ build tools (like `node-gyp` or `make`) that certain NPM packages might rely on.

## Commands Used

```bash
# Example syntax for verifying the optimizations worked locally
docker build -t express-app-optimized .
docker run -p 3000:3000 express-app-optimized

# To verify you are not running as root:
docker exec -it <container_id> whoami
```

## Verification

### Expected Result

You should observe a massive drop in image size when running `docker images` compared to the original unoptimized version. When the container is started, the Express server should still be reachable from your local browser, but it will be running securely as an unprivileged user under the hood.

---

## Challenges Faced

- Ensuring the `chown -R` command targeted the correct directory. Since the original Dockerfile didn't use a `WORKDIR`, files were scattered. Introducing `WORKDIR /usr/src/app` made managing permissions for the non-root user much easier.

---

## Lessons Learned

- Writing an "okay" Dockerfile takes 30 seconds. Writing an "optimal and secure" Dockerfile takes a few extra minutes of thought, but the benefits in production (faster deployments, lower attack surface) are absolutely massive.
- Combining security (non-root users) and size optimization (Alpine images) should be the default starting point for any new containerized application, rather than an afterthought.
