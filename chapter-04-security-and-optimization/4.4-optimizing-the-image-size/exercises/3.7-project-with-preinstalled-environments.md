# Exercise 3.7: Project with preinstalled environments

## Objective

Simplify Dockerfiles and further reduce image sizes by migrating away from bloated standard OS images (like Ubuntu) to lightweight, language-specific Alpine Linux base images.

---

## Concepts Practiced

- Pre-installed Base Images (`node`, `golang`)
- Alpine Linux Variants
- Dockerfile Simplification
- Alpine User Management (`adduser -D`)

---

## Exercise Summary

This exercise continued the optimization of the frontend and backend applications from the previous exercises. Instead of starting from a bare `ubuntu` image and manually running `apt-get` commands to install `curl`, `nodejs`, or `golang`, the `FROM` instructions were updated to use official, pre-configured environments available on Docker Hub. 

Specifically, the `node:16-alpine` and `golang:1.16-alpine` images were chosen. By using these Alpine variants, all the manual package installation commands could be deleted from the Dockerfiles, vastly simplifying them. The frontend image size dropped to around 488 MB, and the backend image settled around 468 MB. While the backend image was slightly larger than the highly-optimized Ubuntu version from the previous exercise, the tradeoff in Dockerfile simplicity and maintainability was significant.

---

## Hints

- Finding the right base image tag is crucial. The frontend specifically required Node 16. Searching Docker Hub for the `node` official image allows you to find the exact tag `16-alpine`.
- When switching from an Ubuntu/Debian base to an Alpine base, remember that the underlying package manager changes from `apt-get` to `apk`.
- The user management commands also change on Alpine! You cannot use `useradd -m`. Instead, you must use `adduser -D` to create your non-root user.
- Since you are using pre-installed images, you no longer need to purge or remove installation files (like `curl`) because the base image maintainers have already optimized the environment for you.

---

## Commands Used

```bash
# Example syntax for pulling a specific alpine variant
docker pull node:16-alpine
docker pull golang:1.16-alpine
```

---

## Verification

### Expected Result

After updating the `FROM` lines and fixing the user creation syntax (`adduser`), the containers should build perfectly. Even though the underlying operating system was completely swapped out (from Ubuntu glibc to Alpine musl), the Node.js and Go applications should run exactly the same when executing `docker compose up`.

---

## Challenges Faced

- Realizing that `useradd` resulted in a "command not found" error during the build process, which required researching how user accounts are created in Alpine Linux.
- Ensuring the exact language versions (Node 16 and Go 1.16) were specified in the image tags to prevent compatibility issues with older source code.

---

## Lessons Learned

- **Don't reinvent the wheel:** If you find yourself writing `RUN apt-get install python3 nodejs golang`, stop and look for an official image that already has those tools installed.
- Official language-specific images (especially their `-alpine` variants) are maintained by experts who have already applied deep optimizations, saving you from writing brittle, manual cleanup scripts.
- Changing a base image can break shell commands within the Dockerfile (like `bash` vs `sh`, or `apt` vs `apk`), so testing the build iteratively is important.
