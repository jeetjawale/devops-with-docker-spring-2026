# Exercise 3.8: Multi-stage frontend

## Objective

Refactor a frontend application's Dockerfile to use a multi-stage build, completely separating the heavy build environment from the final, lightweight production runtime.

---

## Concepts Practiced

- Multi-stage Builds (`AS build-stage`)
- Stage referencing (`COPY --from=...`)
- Applying permissions during copy (`--chown`)
- Eliminating manual cleanup scripts

---

## Exercise Summary

This exercise took the heavily optimized frontend Dockerfile from previous exercises and modernized it using a multi-stage build. In the previous iteration, we had to manually run commands like `rm -rf src node_modules` in a massive, chained `RUN` instruction to keep the image small. 

By implementing a multi-stage build, that hacky cleanup was completely eliminated. The first stage (named `build-stage`) was dedicated purely to building the React application—it copied all the source code, installed all dependencies, and executed `npm run build`. 

The second stage started completely fresh with a new `FROM` instruction. In this clean environment, only the `serve` utility was installed and the non-root user was created. Finally, instead of copying files from the host machine, the compiled static assets were copied directly from the `build-stage` into the new image. Everything else—the source code, the React dependencies, the cache—was automatically discarded.

---

## Hints

- A multi-stage Dockerfile has multiple `FROM` instructions. The last `FROM` instruction dictates the actual final image that is saved to your computer.
- You can alias your first stage by appending `AS <name>` to the `FROM` line (e.g., `FROM node:16-alpine AS build-stage`).
- In your final stage, use the `--from=<name>` flag with the `COPY` instruction to extract files from the previous stage (e.g., `COPY --from=build-stage /usr/src/app/build ./build`).
- Remember the security principles from earlier exercises! When you copy the files into the final stage, you must ensure the non-root user owns them. You can do this efficiently using the `--chown` flag directly on the `COPY` instruction.

---

## Commands Used

```dockerfile
# Example snippet of a multi-stage transition
FROM node:16-alpine AS build-stage
# ... build steps ...

FROM node:16-alpine
# ... setup runtime ...
COPY --from=build-stage --chown=appuser:appuser /path/to/build ./build
```

---

## Verification

### Expected Result

Building the image should execute in two distinct phases visible in the terminal logs. When you run `docker images`, the final size of this multi-stage image should be very similar to (or slightly smaller than) the manually optimized version from Exercise 3.7. However, the Dockerfile itself will be much cleaner, easier to read, and less prone to breaking.

---

## Challenges Faced

- Getting the paths right in the `COPY --from=...` instruction. If you build the application in `/usr/src/app/build` in the first stage, you must explicitly copy from that exact absolute path into the second stage.
- Realizing that `COPY` defaults to `root` ownership, meaning the application would crash with a `Permission denied` error unless `--chown=appuser:appuser` was added to the `COPY` instruction.

---

## Lessons Learned

- Multi-stage builds are the definitive solution to the "bloated container" problem. They eliminate the need for complicated, chained cleanup commands (`&& rm -rf ...`).
- You don't necessarily have to switch base images between stages. Using `node:16-alpine` for the build stage and `node:16-alpine` again for the final stage is perfectly valid, as the primary goal is simply to discard the `node_modules` folder. (Though switching to `nginx:alpine` is also a very popular alternative for serving static frontend files).
